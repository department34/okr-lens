# 3가지 Spec 비교 분석

## 파일별 특징

### 1. new_spec.md (신승희 - 원본/기본)
**특징:**
- 가장 포괄적이고 이론적 기술
- **일정**: 10일 실제 기준 (Day 1~10)
- **범위**: R1 완벽 / R2 반 / R3 간단 구분
- **데이터**: JSON + git 필드 소유권 명시
- **강점**: Day별 구체적 산출물 + 체크리스트 상세
- **약점**: 폴더 구조 미언급, 팀 협력 프로세스 부재

---

### 2. new_spec_중식.md (중식 - 기술 검토 기반)
**특징:**
- **일정**: 3일 집중 개발 + 1주 2회 동기 (Day 1~3 상세)
- **범위**: R1 완벽 > R2 반 정도 > R3 간단하게 (명확한 우선순위)
- **데이터**: shared_kr_data.json 단순화
- **강점**: 
  - 우선순위 명확 (R1 완벽 > R2 반 > R3 간단)
  - 9개 체크 항목 + 위험도 판정 구체적 정의
  - 3주 × 17시간/명 = 51시간 기준 제시
  - 최소 경로 통합(Day 2 오후) 명시
- **약점**: 폴더 구조 부재, 팀 협력 가이드 최소

---

### 3. new_spec_대식.md (대식 - 최종/아키텍처)
**특징:**
- **일정**: 3주 병렬 개발 (Day 1~21 + 협력 프로세스)
- **범위**: R1 완벽 > R2 반 > R3 간단 (중식과 동일)
- **데이터**: 추상화층 (MVP vs 시제품 호환)
- **강점**:
  - 폴더 구조 완전 명시 (r1/r2/r3/shared 분리)
  - 기술 스택 + 아키텍처 상세 (섹션 6)
  - 공유 데이터 계약 정형화 (KRSchema)
  - 팀 협력 가이드 완전 (Git, 코드 리뷰, 동기화, 긴급소통)
  - DB 마이그레이션 로드맵 (시제품 4주차, 26~35시간)
  - 3대 PC 병렬 40분 vs 1대 PC 3시간 명시
- **약점**: Day별 상세 산출물 체크리스트 부족

## 가장 적절한 통합 방향

### 추천: (대식)을 베이스 + (신)(중)의 장점 통합

**1. 일정 기준: 대식의 3주 (현실적)**
   - 병렬 개발 최적화
   - 팀 동기화 포함
   - 하지만 Day별 산출물은 (신)의 체크리스트 참고

**2. 범위 기준: 중식의 우선순위 명시**
   - R1 완벽 > R2 반 > R3 간단 (순서 명확)
   - Feature 범위도 중식의 "이번 MVP" vs "Phase 2" 구분

**3. 데이터 기준: 대식의 추상화층 + 중식의 단순화**
   - dataAccess.js 추상화 (MVP/시제품 호환)
   - shared_kr_data.json 중심 (git 필드소유권 병행)

**4. 폴더 구조: 대식 그대로 (r1/r2/r3/shared)**
   - 병렬 개발 충돌 최소화
   - 각자 독립성 확보

**5. 팀 협력: 대식의 Git workflow + 중식의 간결함**
   - 일일 PR + 주 2회 동기 (대식)
   - 블로커/스키마 변경은 Slack/노션 (대식)
   - 하지만 너무 장황하지 않게 간결화

**6. 기술 스택: 대식 완전 준용**
   - Next.js 14, Tailwind, Recharts
   - Claude API (R1 상시, R2/R3 선택)
   - shared/diagnosticEngine.js (위험 태깅 규칙)
