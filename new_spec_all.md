# OKR Lens — 통합 MVP Spec (최종 완성본 v2.0)

> **최종 통합본** | 2026-06-24 | 신승희(기본) + 박중식(우선순위·규칙) + 곽대식(아키텍처·협력) 통합
> 
> **3개 Spec 통합**: 신승희(범위·Day 상세) + 중식(우선순위·엔진) + 대식(아키텍처·협력) 최적 조합
>
> **개발 기간**: 3주 (21일) 병렬 개발 | **팀**: 3명 동시 (R1 완벽 > R2 반 > R3 간단)
>
> **핵심 개선**: 폴더 분리 + 팀 협력 프로세스 + Day별 체크리스트 + 우선순위 명확화 + 데이터 계약 정형화

---

## 0. 개요 · 통합 · 시연 연계 (3역할 공통)

**제품 한 줄**: 2026 OKR 기반 개인평가제도를 처음 운영하는 IT 조직을 돕는 3역할 도구 — KR을 잘 쓰게(피평가자) → 잘 검토·코칭·승인하게(평가자) → 전사 품질·위험을 보게(인사담당자).

**3역할 / 동시 개발**: R1 피평가자 · R2 평가자 · R3 인사담당자를 **담당자 1인씩, 3명이 동시 개발**.
- **우선순위**: R1 완벽 > R2 반 정도 > R3 간단하게

**시연 스토리 (핸드오프)**:
1. **R1 피평가자** — 업무·제약·협업(상위 Objective는 선택) 입력 → AI가 KR 초안 4종 생성 → 대화형으로 정제 → 확정·제출.
2. **R2 평가자** — 제출된 KR을 받아 규칙 위반 자동 점검(AI Validation) → 승인.
3. **R3 인사담당자 (클로징)** — 승인된 KR 포함 전사 KR을 집계 → 이의제기 위험 KR 사전 식별 → 업무군 도출·확정 → 구성·난이도/품질 분포 제시. 

**공유 KR 데이터 계약** (3역할 공통 스키마):
```json
{
  "id": "KR_2026_001",
  "담당자": "박중식",
  "부서": "개발팀",
  "직급": "개발자",
  "objective": "OKR 기반 평가제도 구축",
  "kr": "KR 작성·정제 Copilot 구현으로 팀원 작성 자신감 향상",
  "측정방법": "SMART 기준: 초안 4종 생성 → 대화형 정제 → 최종 KR 확정",
  "체크": [1, 0, 1, 1, 1, 0, 1, 1, 1],
  "난이도": "상",
  "평가자": "곽대식",
  "승인상태": "제출|승인|반려",
  "승인일시": "2026-07-10T14:30:00",
  "위험도": "중",
  "위험유형": ["측정모호", "신기술의존"],
  "업무군_확정": "개발/요건",
  "생성일시": "2026-06-25T09:00:00"
}
```

**데이터 흐름**:
- R1: KR 입력 → 생성 → 정제 → 확정 → **shared_kr_data.json에 저장** (승인상태=제출)
- R2: shared_kr_data.json에서 KR 읽기 → AI Validation → 승인/반려 → **파일 업데이트** (승인상태, 위험도, 위험유형)
- R3: shared_kr_data.json에서 승인된 KR 읽기 → 위험도 필터링 → 사전 코칭 리스트 제시

**공유 진단 엔진**: 위험 태깅 규칙은 R2(승인 시 위반 점검)와 R3(전사 위험 식별)가 **공유** — 판정 일관·중복 구현 방지.

**구동 환경**: **로컬 서버(localhost) 구동**을 기본. **Next.js 14 단일 앱**에 역할별 라우팅(`/r1-employee`·`/r2-evaluator`·`/r3-hr`). 한 번의 `npm run dev`로 3역할을 모두 띄워 시연 핸드오프가 자연스럽다.

## 1. 문제 — 누구의 어떤 불편을 푸는가

**R1 피평가자**: OKR 운영 경험은 있으나 KR 문장을 직접 쓰는 자신감이 부족하다. 무엇을 입력해 시작할지, 자기 KR이 측정 가능한지·도전적인지 스스로 판단하기 어렵고, 고치려 해도 어떻게 다듬을지 막막하다.

**R2 평가자(팀장)**: OKR 작성 기준(운영/전략혁신, 난이도 체크 9항목, KR 4~6개 등)이 복잡해 승인 시 규칙 위반을 즉각 판단하기 어렵다. 코칭 기록을 의무 등록해야 하나 무엇을 남길지 막막하다.

**R3 인사담당자**: 전 직원 KR을 일관 기준으로 검토·코칭해야 하나 인력은 1명이라 형식적 검토에 그친다. 평가위원회 전 이의제기 위험 KR을 미리 거를 도구가 없다.

## 2. 사용자 — 핵심 페르소나

| 역할 | 페르소나 | 상황·목표 |
|------|----------|-----------|
| **R1 피평가자** | 자기 KR을 작성하는 직원 (예: HR기획팀 작성자) | 입력만 하면 초안을 받고, 대화형 정제로 측정 가능·도전·정렬된 KR로 다듬어 자신 있게 제출 |
| **R2 평가자** | 박준혁 (팀장, 책임급, IT서비스개발팀) | 첫 OKR 제도. 규칙 위반을 자동으로 감지해서 공정·근거 있게 승인하고 싶음 |
| **R3 인사담당자** | 신승희 (인사노무팀, 과장급, 제도 단독 설계·운영) | 위험 KR을 사전에 식별해 코칭하고, 업무군 구성·난이도 분포로 평가자에게 참고자료 제공 |

## 3. 핵심 기능 (MVP) — 역할별 우선순위

> **우선순위 명확화**: **R1 완벽** ✅ > **R2 반 정도** ⚡ > **R3 간단하게** 🔹

### R1 피평가자 — AI KR 작성·정제 Copilot (완벽) ✅

작성자가 KR을 처음부터 끝까지 만들고 다듬어 평가자에게 제출하는 흐름.

**Feature 1. KR 초안 입력 → AI 4종 초안 생성**
- 업무 설명 · 제약 조건 · 협업 대상을 입력 → AI가 현실적인 KR 초안 **4가지** 생성
- 상위 Objective는 선택 입력 (없어도 됨)

**Feature 2. KR 대화형 정제**
- "데이터 품질 기준을 더 구체적으로", "목표를 90점으로 올려 더 도전적으로" 같은 **자연어 피드백** → AI가 수정안 + 개선 포인트 제시
- "다른 제안 보기 / 이 내용으로 적용" 선택지 제공
- 3~5회 반복 가능

**Feature 3. 최종 검토 — 단일 모델/룰 기반 검토**
- MVP는 **단일 모델(Claude) 또는 룰 기반 단일 검토**로 구현
- ※ 멀티 AI(Claude·Gemini·OpenAI) 동시 교차 검증 → Phase 2

**Feature 4. 확정 → 평가자 제출**
- 최종 KR 편집·확정 → **shared_kr_data.json에 저장** (승인상태=제출)
- R2가 읽을 수 있게 준비

> **MVP 완성 대상**: Feature 1·2·4 모두 완성. Feature 3은 축소.

---

### R2 평가자 — OKR Lens (반 정도) ⚡

**Feature 1. 메인 대시보드 (극단적으로 간단하게)**
- 이번 달 OKR 승인률 카드 **1~2개만** (캘린더·랭킹 제외)

**Feature 2. OKR 검토·승인**
- 팀원 OKR 아코디언 뷰 (Objective → KR)
- **AI Validation 버튼**: 9개 체크 항목 자동 평가 → 위험도·위험유형 산출 + 위반 항목 텍스트 출력
- 승인 / 반려(사유 입력) → shared_kr_data.json 업데이트
- 코칭 메모 수기 입력 (AI 자동 제안 제외)

> **MVP 완성 대상**: Feature 2(검토·승인 + AI Validation)만 완성. Feature 1은 극단적으로 간단.

---

### R3 인사담당자 — 전사 OKR 위험 식별 (간단하게) 🔹

**Feature 1. 이의제기 위험 KR 자동 식별 → 사전 코칭 리스트 (필수)**
- 전 직원 KR 로드 시 위험 태깅 엔진이 KR별 **위험도·위험유형** 자동 추가
- 위험도(상/중/하)·유형·부서 **필터·정렬** → "사전 코칭 대상" 리스트
- KR 클릭 시 상세 패널: 원문 + 9개 체크 항목 + 위험 유형 + 권고 1줄
- **위험 태그 = "코칭 후보", 위반 아님** — 확정 아님, 사전 점검용

**Feature 2. 업무군 도출·확정 (선택)**
- 1차 도출: 엔진이 KR 분석해 업무군 자동 분류 (9가지 시드 + 경계/미분류)
- 인사담당자가 추가·재지정 후 "확정"
- 확정 분류로 부서별 **업무군별 지표 건수·비율** 시각화

**Feature 3. 난이도·품질 분포 (여유 있으면)**
- 난이도 상/중/하 분포 (전사·부서별)
- 측정가능성·도전성 2축 분포
- ※ 시간 부족하면 제외 가능

> **MVP 완성 대상**: Feature 1(위험 리스트) 필수. Feature 2·3은 여유 있을 때.

## 4. 제외 범위 — 이번엔 안 하는 것

**공통**
- 사용자 인증·권한 관리 (프로토타입 단일 계정)
- 실제 DB·인사시스템 연동 (JSON mock 대체)
- 이메일/슬랙 알림 (앱 내 배지·리스트로 대체)

**R1 피평가자**
- 멀티 AI 교차 검증 → Phase 2
- 품질 진단 실시간 3축 점수 화면 → Phase 2
- 작성 이력 영구 저장 (세션 내 임시만)

**R2 평가자**
- ⛔ 팀원 실시간 랭킹 테이블 · 이모지 등급 · 최하 팀원 알림 → Phase 2
- 메인 대시보드 캘린더 → Phase 2
- AI 코칭 자동 제안 (수기만) → Phase 2
- 연말 평가 점수 입력

**R3 인사담당자**
- 정렬성(상위 목표 정렬) → 스펙 전체 제외 (팀 OKR 미정의)
- 난이도 일관성 자동 검증 → Phase 2
- 비지도 업무군 자동 군집 → Phase 2

## 7. 위험 태깅 엔진 — 9개 체크 항목 & 위험도 판정

### 7.1 9개 체크 항목 (각 Yes=1, No=0)

```
① 수치로 측정 가능한가?
   ✅ "매출 10% 증가" / ❌ "고객 만족도 개선" 모호

② 외부 의존성이 없는가? (팀이 통제 가능한가?)
   ✅ "팀이 달성 가능" / ❌ "날씨에 따라 달라짐"

③ 도전적인가? (유지형이 아닌가?)
   ✅ "80점→90점" / ❌ "현 상태 유지"

④ 시간 내에 현실적으로 달성 가능한가?
   ✅ "3개월 안에 가능" / ❌ "불가능해 보임"

⑤ 명확한 언어인가? (팀원이 이해할 수 있는가?)
   ✅ "서버 응답시간 50% 단축" / ❌ "성능 개선" 모호

⑥ 다른 팀 KR과 겹치지 않는가?
   ✅ "우리팀만의 고유한 KR" / ❌ "다른팀도 같은 KR"

⑦ 신기술/새 도구에 과도하게 의존하지 않는가?
   ✅ "배우지 않은 것 1개 정도" / ❌ "3개 이상 새로운 것"

⑧ 단순 건수형이 아닌가? (질이 있는가?)
   ✅ "고객 5명 만남 + 피드백 반영" / ❌ "미팅 10회만"

⑨ 평가자가 확인할 수 있는가? (증거가 있는가?)
   ✅ "대시보드 수치로 확인 가능" / ❌ "주관적 판단"
```

### 7.2 위험도 판정

```

0~1개                      →  🟢 '하' (코칭 불필요)
2~3개                      →  🟡 '중' (주의, 보완 권고)
4개 이상                   →  🔴 '상' (높은 위험, 사전 코칭)
```
