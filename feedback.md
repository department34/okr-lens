## 🔧 규칙 엔진 분석 & 검증

### 업무군 분류 규칙 (매우 명확)

**시드 9종** (A등급 산출물·측정 수치로 판단):

1. 개발/요건 (신규 구현)
2. 데이터/정합성 (대사·매핑·현행화·집계·인터페이스·DB)
3. 자동화 (수작업→자동)
4. 성능/튜닝 (응답·처리시간·속도·SQL·CPU)
5. 장애/운영안정 (비정상종료·오류율·정상종료율·모니터링)
6. 정시/납기/처리율 (일정준수·CSR/WOR·조치 비율·리드타임)
7. 매뉴얼/표준/문서 (매뉴얼·가이드·체크리스트·교육)
8. 보안/권한/개인정보 (점검·인증·심사·취약점)
9. 고객/사업기여/제안 (선제안·만족도·아이디어·M/M·매출)

**판단 우선순위**:

- A등급 최종 산출물 읽기 → 실제 측정 수치 식별 → 시드 매칭
- 부서명·도메인만으로 분류 금지
- 미분류 → '경계'로 표시 (주업무군 1개 지정)

### 위험 태깅 규칙 (패턴 기반)

**7가지 쟁점 유형** (자동 감지 가능):

1. **사용불가(C01)**: 자격·교육 이수가 KR/성과로 둔갑 → 정규식: "자격|교육|이수|수료"
2. **현상유지**: 가동률·정상종료율 "유지"형 → 정규식: "유지|0건|정상|가동"
3. **통제밖(C11)**: SLA·고객만족·매출·외부평가 → 정규식: "SLA|고객|매출|외부"
4. **건수형(C06)**: 채택·효과 없이 "건수"만 → 정규식: "건수|개|건|채택|건의"
5. **측정모호(C05)**: 정성 수식어·측정 부재 → 정규식: "개선|높이다|강화|활성화" (정성어)
6. **도전성 저하**: 유지형 KR → 현상유지와 중복
7. **공모형**: 선제안 등 전사 공통지표 (사전에 리스트 정의)

**위험도 산정**:

- 태그 1개 = 위험도 "하"
- 태그 2개 이상 = 위험도 "중"
- C01/C11 포함 = 위험도 "상"

### 구현 복잡성 평가

✅ **복잡성 낮음** (예상 1~2일):

- 선택지가 명확 (9가지 시드 + 경계/미분류)
- 판단 논리가 순차적 (읽기 → 식별 → 매칭)
- 쟁점 태그는 키워드·정규식 기반
- JS로 약 200~300줄 모듈화 가능

⚠️ **주의사항**:

- 규칙 정의는 명확하나, **실제 데이터로 5~10개 케이스 검증 필수** (예외 케이스 확인)
- **A등급 산출물 읽는 휴리스틱**이 정확해야 함 (사람 경험 필요)

## 🤝 협업 전략 (상세)

### Phase 0: 첫 PR 경험 (Day 0~1 — 2~3시간)

**목표**: git 브랜치·PR·충돌·merge를 몸으로 배우기

```
1. 환경 셋업 (30분)
   - 각자 repo clone
   - npm install
   - npm run dev 확인

2. 함께 첫 PR 하기 (90분)
   - 3명이 동시에 main에서 feature/setup 브랜치 생성
   - 각자: src/pages/employee.jsx | evaluator.jsx | hr.jsx 생성
   - 간단한 "Hello [Role]" 페이지 작성
   - develop 브랜치에 PR 제출 (각각)
   - 1번 PR은 merge, 2번 PR부터 충돌 해결
   - merge 후 git pull하며 다른 코드 확인

3. 질문·정리 (30분)
   - git Q&A: "언제 pull?", "브랜치 자동 삭제?", "merge vs rebase?" 등
```

### Phase 1: 역할별 병렬 개발 (Day 3~12)

**브랜치 전략**:

```
main (보호됨, PR만 머지)
  ↓
develop (통합 브랜치, 매일 sync)
  ├─ feature/r1-kr-generation (박중식)
  ├─ feature/r2-coaching-dashboard (곽대식)
  └─ feature/r3-risk-analysis (신승희)
```

**일일 프로세스**:

```
[매일 아침 10분 — 슬랙/회의]
- 어제 뭘 했고, 오늘 뭘 하고, 막힌 게 뭔지 공유

[개인 작업 3시간 50분]
- 자신의 feature 브랜치에서 개발

[매일 저녁 10분]
- 기능 완성되면 develop에 PR 제출
- 다른 2명이 "데이터 연계 부분 맞나?" 5분 코드리뷰
- approve 후 자동 머지 (또는 수동)

[주 2회 정기점검 — 월·목 저녁 30분]
- 3명 함께 통합 테스트 (R1→R2→R3 데이터 흐름)
- 스키마 변경 필요하면 긴급 조정
```

### 충돌 최소화 전략

1. **공유 데이터 계약을 Day 1 오전에 고정** → 이후 수정 금지
2. **각 역할이 다른 파일만 수정**
   - src/pages/employee.jsx (R1 전담)
   - src/pages/evaluator.jsx (R2 전담)
   - src/pages/hr.jsx (R3 전담)
3. **공유 파일은 한 사람만 수정**
   - utils/diagnostic.js (규칙 엔진) → R1이 초안, R2·R3이 PR로 요청
   - utils/businessUnit.js (업무군 분류) → R3 전담

### 연계 검증 체크리스트

**Day 6 정기점검** (R1→R2 넘김):


- **함의**: "기존 제도가 결함" ❌ → "새 기준으로 보완" ✓
