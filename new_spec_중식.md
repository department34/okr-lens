> 개선본 v2.0 | 2026-06-24 | 기술 검토 기반 범위 조정 & 구체화
> 원본: 통합 v1.4 (2026-06-22)
> 변경: 범위 재정리(R1 완벽/R2 반/R3 간단) + 위험 태깅 규칙 9항목 정의 + 데이터 흐름 구체화 + Day별 산출물 명시

## 0. 개요 · 통합 · 시연 연계 (3역할 공통)

**제품 한 줄**: 2026 OKR 기반 개인평가제도를 처음 운영하는 IT 조직을 돕는 3역할 도구 — KR을 잘 쓰게(피평가자) → 잘 검토·승인하게(평가자) → 위험을 사전 식별하게(인사담당자).

**3역할 / 동시 개발**: R1 피평가자 · R2 평가자 · R3 인사담당자를 **담당자 1인씩, 3명이 동시 개발**. 우선순위: **R1 완벽 > R2 반 정도 > R3 간단하게**.

**시연 스토리 (핸드오프)**:
1. **R1 피평가자** — 업무·제약·협업 입력 → AI가 KR 초안 4종 생성 → 대화형으로 정제 → 확정·제출.
2. **R2 평가자** — 제출된 KR을 받아 규칙 위반 자동 점검(AI Validation) → 승인.
3. **R3 인사담당자** — 승인된 KR 포함 전사 KR에서 이의제기 위험 KR 자동 식별 → 사전 코칭 리스트 제시.
   - R1·R2에서 생성된 KR을 R3 화면에서 보여줌 → **시연 핸드오프 완성**.

**공유 KR 데이터 계약** (3역할 공통 스키마):
```json
{
  "id": 1,
  "부서": "IT서비스개발팀",
  "담당자": "김철수",
  "직급": "주임",
  "objective": "데이터 기반 의사결정 고도화",
  "kr": "데이터 파이프라인 지연시간 50% 단축",
  "측정방법": "대시보드 지연시간 지표",
  "달성도": null,
  "난이도": 4,
  "체크": [1, 0, 1, 1, 1, 0, 1, 1, 1],
  "평가자": "박준혁",
  "승인상태": "승인",
  "위험도": "중",
  "위험유형": ["측정모호", "신기술의존"]
}
```

**데이터 흐름**:
- **저장 위치**: `data/shared_kr_data.json` (3역할이 모두 읽고 씀)
- **R1**: KR 입력 → 생성 → 정제 → 확정 → **shared_kr_data.json에 저장** (승인상태=제출, 체크=초기값)
- **R2**: shared_kr_data.json에서 KR 읽기 → AI Validation(위험 태깅) → 승인/반려 → **파일 업데이트** (승인상태, 위험도, 위험유형 추가)
- **R3**: shared_kr_data.json에서 승인된 KR 읽기 → 위험도 필터링 → 사전 코칭 리스트 조회.

**공유 진단 엔진** (위험 태깅): R2 승인 시 + R3 조회 시 동일하게 호출되는 `engine.위험태깅()` 함수 (9개 체크 항목 → 위험도·유형 산출).

**구동 환경**: **로컬 서버(localhost) 구동**을 기본. Next.js 14 단일 앱, 역할별 라우팅(`/employee`·`/evaluator`·`/hr`). 한 번의 `npm run dev`로 3역할을 모두 띄워 시연.

## 1. 문제 — 누구의 어떤 불편을 푸는가

**R1 피평가자**: OKR 운영 경험은 있으나 KR 문장을 직접 쓰는 자신감이 부족하다. 무엇을 입력해 시작할지, 자기 KR이 측정 가능한지·도전적인지 스스로 판단하기 어렵다.

**R2 평가자(팀장)**: OKR 작성 기준(운영/전략혁신, 난이도 체크 9항목, KR 4~6개 등)이 복잡해 승인 시 규칙 위반을 즉각 판단하기 어렵다.

**R3 인사담당자**: 전 직원 KR을 일관 기준으로 검토해야 하나 인력은 1명이라 형식적 검토에 그친다. 평가위원회 전 이의제기 위험 KR을 미리 거를 도구가 없다.

## 2. 사용자 — 핵심 페르소나

| 역할 | 페르소나 | 상황·목표 |
|------|----------|-----------|
| **R1 피평가자** | 자기 KR을 작성하는 직원 | 입력만 하면 초안을 받고, 대화형 정제로 측정 가능·도전·정렬된 KR로 다듬어 제출하고 싶음 |
| **R2 평가자** | 박준혁 (팀장, IT서비스개발팀) | 규칙 위반을 자동으로 감지해서 공정하게 승인하고 싶음 |
| **R3 인사담당자** | 신승희 (인사노무팀, 과장급) | 위험 KR을 사전에 식별해 코칭하고, 시연 때 구성·분포를 제시하고 싶음 |

## 3. 핵심 기능 (MVP) — 역할별 우선순위

### **R1 피평가자 — AI KR 작성·정제 Copilot (완벽)** ✅

**Feature 1. KR 초안 입력 → AI 4종 초안 생성**
- 업무 설명 · 제약 조건 · 협업 대상을 입력하면(상위 Objective는 선택) AI가 현실적인 KR 초안 **4가지**를 생성.

**Feature 2. KR 대화형 정제**
- "더 도전적으로", "측정방법 구체화" 같은 **자연어 피드백** → AI가 수정안 + 개선 포인트 제시.

**Feature 3. 확정 → 평가자 제출**
- 최종 KR을 편집·확정하면 공유 KR 데이터(승인상태=제출)로 **저장** → R2가 읽을 수 있게.

> MVP 핵심: Feature 1·2·3 모두 완성.

---

### **R2 평가자 — OKR Lens (반 정도)** ⚡

**Feature 1. 메인 대시보드 (간단하게)**
- 이번 달 OKR 승인률 카드 **1~2개만**. (캘린더 제외)

**Feature 2. OKR 검토·승인**
- 팀원 OKR 아코디언 뷰(Objective → KR).
- **AI Validation 버튼**: 2026 제도 기준 위반 자동 감지 (9개 체크 항목, 위험도 판정).
- 승인 / 반려(사유 입력) → 상태 저장.
- 코칭 메모 수기 입력 (AI 자동 제안 제외).

> MVP 핵심: Feature 2(검토·승인 + AI Validation) 완성. Feature 1은 극단적으로 간단하게.

---

### **R3 인사담당자 — 전사 OKR 위험 식별 (간단하게)** 🔹

**Feature 1. 이의제기 위험 KR 자동 식별 → 사전 코칭 리스트**
- 전 직원 KR 로드 시 위험 태깅 엔진이 KR별 **위험도·위험유형**을 자동 추가.
- 위험도(상/중/하)·유형·부서 **필터·정렬** 한 "사전 코칭 대상" 리스트 + 상세 패널.
- 상세 패널: 원문 + 9개 체크 항목 + 위험 유형 + 권고 1줄.

> MVP 핵심: Feature 1만 완성. 업무군·분포는 Phase 2.

## 4. 제외 범위 — 이번엔 안 하는 것

**공통**
- 사용자 인증·권한 관리 (프로토타입 단일 계정)
- 실제 DB·인사시스템/ERP 연동 (JSON mock 대체)
- 이메일/슬랙 알림 (앱 내 배지·리스트로 대체)

**R1 피평가자**
- 최종 검토(멀티 AI 교차 검증) → Phase 2
- 품질 진단(실시간 3축 점수) → Phase 2
- 작성 이력 영구 저장 (세션 내 임시 저장만)

**R2 평가자**
- 메인 대시보드 캘린더 → Phase 2
- AI 코칭 자동 제안 (수기 입력만)
- 팀원 랭킹·이모지 등급 → Phase 2
- 연말 평가 점수 입력

**R3 인사담당자**
- 업무군 도출·확정 UI → Phase 2
- 난이도·품질 분포 대시보드 → Phase 2
- 정렬성(상위 목표 정렬) → 스펙 제외 (팀 OKR 미정의)

## 5. 성공 기준 — 시연에서 무엇이 되면 성공인가

**R1 피평가자**
- [ ] 업무·제약·협업을 입력하면 KR 초안 4종이 생성된다
- [ ] 자연어 피드백으로 KR이 정제된다(수정안 + 개선 포인트)
- [ ] 확정하면 KR이 shared_kr_data.json에 저장된다 (승인상태=제출)

**R2 평가자**
- [ ] OKR 카드에서 AI Validation 버튼을 누르면 9개 체크 항목이 자동 계산되고 위험도가 출력된다
- [ ] 승인/반려 동작이 작동하고 shared_kr_data.json이 업데이트된다
- [ ] 코칭 메모를 입력할 수 있다

**R3 인사담당자**
- [ ] 전사 KR(shared_kr_data.json)을 로드하면 KR별 위험도·위험유형이 자동 태깅된다
- [ ] 위험도 필터(상/중/하)가 작동한다
- [ ] 상세 패널에서 9개 체크항목·위험유형·권고가 보인다

**통합 (핸드오프)**
- [ ] R1이 작성·제출 → R2가 승인한 KR이 R3 위험 리스트에 반영되어 나타난다

```

             (/employee=R1, /evaluator=R2, /hr=R3)
스타일링:    Tailwind CSS
언어:        JavaScript (JSX)
상태관리:    React useState / useContext + localStorage
데이터:      공유 KR 데이터셋(JSON): data/shared_kr_data.json
진단 엔진:   룰 기반 JS 모듈 — 위험 태깅 (9개 체크 → 위험도·유형)
             src/lib/engine.js: engine.위험태깅(KR) → { 위험도, 위험유형[] }
AI:          Anthropic Claude API (claude-sonnet-4-6)
             — R1: KR 초안 생성·대화형 정제 (상시 사용)
             — R2·R3: 핵심 진단은 룰 기반(API 불필요)
구동·배포:   로컬 서버(npm run dev) 또는 Vercel
환경변수:    .env.local (ANTHROPIC_API_KEY, 선택)
```

## 7. 위험 태깅 엔진 — 9개 체크 항목 & 위험도 판정

### **9개 체크 항목** (각 Yes=1, No=0)

```
① 수치로 측정 가능한가?
   (✅ "매출 10% 증가" / ❌ "고객 만족도 개선" 모호)

② 외부 의존성이 없는가? (팀이 통제 가능한가?)
   (✅ "팀이 달성 가능" / ❌ "날씨에 따라 달라짐")

③ 도전적인가? (유지형이 아닌가?)
   (✅ "80점→90점" / ❌ "현 상태 유지")

④ 시간 내에 현실적으로 달성 가능한가?
   (✅ "3개월 안에 가능" / ❌ "불가능해 보임")

⑤ 명확한 언어인가? (팀원이 이해할 수 있는가?)
   (✅ "서버 응답시간 50% 단축" / ❌ "성능 개선" 모호)

⑥ 다른 팀 KR과 겹치지 않는가?
   (✅ "우리팀만의 고유한 KR" / ❌ "다른팀도 같은 KR")

⑦ 신기술/새 도구에 과도하게 의존하지 않는가?
   (✅ "배우지 않은 것 1개 정도" / ❌ "3개 이상 새로운 것")

⑧ 단순 건수형이 아닌가? (질이 있는가?)
   (✅ "고객 5명 만남 + 피드백 반영" / ❌ "미팅 10회만")

⑨ 평가자가 확인할 수 있는가? (증거가 있는가?)
   (✅ "대시보드 수치로 확인 가능" / ❌ "주관적 판단")
```

### **위험도 판정**

```
위반 항목 수 (체크=0인 항목)  →  위험도
0~1개                      →  🟢 위험도 '하' (코칭 불필요)
2~3개                      →  🟡 위험도 '중' (주의, 보완 권고)
4개 이상                   →  🔴 위험도 '상' (높은 위험, 사전 코칭)
```

### **위험 유형** (어떤 항목이 위반되었는지)

```
위반 항목 → 위험 유형
①         → "측정모호"
②         → "통제불가"
③         → "도전성저하"
④         → "현실성낮음"
⑤         → "표현모호"
⑥         → "공모형"
⑦         → "신기술과의존"
⑧         → "건수형지표"
⑨         → "확인불가"
```
