## ✅ 스펙 평가

### 잘한 점

1. **문제 정의가 역할별로 매우 구체적입니다.**
   - 막연히 "OKR이 어렵다"가 아니라 행동 단위로 풀었습니다
   - R1: 「무엇을 입력해 시작할지, 자기 KR이 측정 가능한지·도전적인지 판단하기 어렵다」
   - R2: 「OKR 기준이 복잡해 룰 위반을 즉각 판단하기 어렵다」
   - R3: 「1명 인력으로 형식적 검토에 그친다」 ← **핵심 제약까지 짚음**

2. **범위 통제가 이례적으로 좋습니다.**
   - 「멀티 AI는 외부 다중 API, 폐쇄망·MVP 범위 밖 → Phase 2」로 명확히 미룸
   - 「⛔ 랭킹 테이블·이모지 등급 제외」 — 욕심나는 기능을 스스로 뺌
   - **이 팀의 범위 통제가 다른 팀의 참고가 될 정도**

3. **3역할 분리 설계가 우수합니다.**
   - 「각 역할은 자기 화면만, 다른 역할과는 데이터로만 연계」
   - 진단 엔진은 「R2·R3이 공유 → 판정 일관·중복 구현 방지」
   - 3명 병렬 개발에 최적화된 구조

4. **개념 정리가 정확합니다.**
   - 「위험 태그 = '코칭 후보(이상 징후 트리거)', 위반 아님」
   - 소송 리스크를 미리 고려해 톤을 명확히 함

### 개선하면 좋은 점

1. **일정이 3주 해커톤과 맞지 않습니다 (⭐ 중요)**
   - **현재**: Day 1/2/3에 모든 것 끝내려는 계획
   - **문제**: 3명이 각자 LLM·규칙엔진·대시보드를 3일에 끝내려니 리스크 큼
   - **해결**: **10~15일을 충분히 사용**. Day 1~3에 1차 기능 완성 후, 남은 약 2주를 통합·정제·시연 안정화에 배정

2. **통합이 가장 늦게 연결됩니다 (⭐ 가장 중요)**
   - **현재**: 통합 리허설이 Day 3의 마지막
   - **문제**: 이 팀의 승부처는 「R1→R2→R3 한 줄기 흐름」인데, 마지막에야 시도함
   - **해결**: **KR 1건이라도 R1→R2→R3까지 끝까지 흐르는 최소 경로를 Day 3~5에 먼저 통과**시켜 공유 스키마 검증. 그 위에 각 역할 고도화 추가

3. **R3 담당자의 부하가 가장 큽니다 (⭐ 부하 관리)**
   - **현재**: 위험 태깅 + 업무군 도출·확정 + 난이도·품질 분포를 혼자 담당
   - **해결**: 
     - **우선순위**: Feature 1(위험 태깅) > Feature 2(업무군 확정) >> Feature 3(분포 차트)
     - 시간 부족하면 Feature 3은 후순위로 두고 Feature 1·2를 완성하는 게 낫음

### 업무군 분류 규칙 (매우 명확)

**시드 9종** (A등급 산출물·측정 수치로 판단):
1. 개발/요건 (신규 구현)
2. 데이터/정합성 (대사·매핑·현행화·집계·인터페이스·DB)
3. 자동화 (수작업→자동)
4. 성능/튜닝 (응답·처리시간·속도·SQL·CPU)
5. 장애/운영안정 (비정상종료·오류율·정상종료율·모니터링)
6. 정시/납기/처리율 (일정준수·CSR/WOR·조치 비율·리드타임)
7. 매뉴얼/표준/문서 (매뉴얼·가이드·체크리스트·교육)
8. 보안/권한/개인정보 (점검·인증·심사·취약점)
9. 고객/사업기여/제안 (선제안·만족도·아이디어·M/M·매출)

**판단 우선순위**:
- A등급 최종 산출물 읽기 → 실제 측정 수치 식별 → 시드 매칭
- 부서명·도메인만으로 분류 금지
- 미분류 → '경계'로 표시 (주업무군 1개 지정)

### 위험 태깅 규칙 (패턴 기반)

**7가지 쟁점 유형** (자동 감지 가능):
1. **사용불가(C01)**: 자격·교육 이수가 KR/성과로 둔갑 → 정규식: "자격|교육|이수|수료"
2. **현상유지**: 가동률·정상종료율 "유지"형 → 정규식: "유지|0건|정상|가동"
3. **통제밖(C11)**: SLA·고객만족·매출·외부평가 → 정규식: "SLA|고객|매출|외부"
4. **건수형(C06)**: 채택·효과 없이 "건수"만 → 정규식: "건수|개|건|채택|건의"
5. **측정모호(C05)**: 정성 수식어·측정 부재 → 정규식: "개선|높이다|강화|활성화" (정성어)
6. **도전성 저하**: 유지형 KR → 현상유지와 중복
7. **공모형**: 선제안 등 전사 공통지표 (사전에 리스트 정의)

**위험도 산정**:
- 태그 1개 = 위험도 "하"
- 태그 2개 이상 = 위험도 "중"
- C01/C11 포함 = 위험도 "상"

### 구현 복잡성 평가

✅ **복잡성 낮음** (예상 1~2일):

**목표**: git 브랜치·PR·충돌·merge를 몸으로 배우기

```
1. 환경 셋업 (30분)
   - 각자 repo clone

2. 함께 첫 PR 하기 (90분)
   - 3명이 동시에 main에서 feature/setup 브랜치 생성
   - 각자: src/pages/employee.jsx | evaluator.jsx | hr.jsx 생성

   - develop 브랜치에 PR 제출 (각각)

   - merge 후 git pull하며 다른 코드 확인

3. 질문·정리 (30분)
   - git Q&A: "언제 pull?", "브랜치 자동 삭제?", "merge vs rebase?" 등

### Phase 1: 역할별 병렬 개발 (Day 3~12)

**브랜치 전략**:
```
main (보호됨, PR만 머지)
  ↓
develop (통합 브랜치, 매일 sync)
  ├─ feature/r1-kr-generation (박중식)
  ├─ feature/r2-coaching-dashboard (곽대식)
  └─ feature/r3-risk-analysis (신승희)

**일일 프로세스**:
```

[개인 작업 3시간 50분]
- 자신의 feature 브랜치에서 개발

[매일 저녁 10분]
- 기능 완성되면 develop에 PR 제출
- 다른 2명이 "데이터 연계 부분 맞나?" 5분 코드리뷰
- approve 후 자동 머지 (또는 수동)

[주 2회 정기점검 — 월·목 저녁 30분]
- 3명 함께 통합 테스트 (R1→R2→R3 데이터 흐름)

```

## ⚠️ 주요 주의사항

### 1. 소송·표현 제약 (최우선)

- **톤**: "위반/잘못" ❌ → "코칭·기준 통일·연속 개선" ✓
- **문구**: "이의제기 위험" ❌ → "코칭 후보(이상 징후 트리거)" ✓
- **함의**: "기존 제도가 결함" ❌ → "새 기준으로 보완" ✓
- **일관성**: 모든 UI·AI 생성 문구·데이터베이스에 적용 필수

**체크리스트**:
```

□ R3 위험 태그 UI에서 "위반" 아니라 "코칭 후보" 표시
□ mock 데이터의 코칭문구에 톤 적용
□ 발표 스크립트 리뷰 (법무팀?)
```

### 2. 규칙 엔진 검증 필수

- **구현 후 샘플 테스트**: 실제 KR 5~10개로 위험 태그·업무군 분류 검증 (Day 9)

**Day 9 체크**:
