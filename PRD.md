## 📋 Executive Summary

**OKR Lens**는 2026 OKR 기반 개인평가제도를 처음 운영하는 IT 조직을 지원하는 3역할 협력 플랫폼입니다.

- **피평가자(R1)**: AI 기반 KR 초안 자동 생성 → 대화형 정제 → 자신감 있게 제출
- **평가자(R2)**: AI 규칙 검증 → 근거 있는 승인/반려 → 공정한 코칭
- **인사담당자(R3)**: 위험 KR 자동 식별 → 사전 코칭 → 평가위원회 사전 준비

**핵심 가치**: 불완전한 KR로 인한 평가 분쟁 사전 방지 + 실무자의 KR 작성 자신감 향상

## 🎭 Core Narrative

### 한 조직, 세 역할의 갇힌 상황

2026년 처음으로 OKR 기반 개인평가제도를 도입한 IT 조직. 하지만 세 역할이 각자 다른 문제에 갇혀 있다.

**R1 피평가자** — 직원은 OKR 경험이 있지만, KR을 어떻게 작성해야 할지 자신감이 없다. 측정 가능한지, 도전적인지, 현실적인지 판단하기 어려우므로 매번 팀장에게 물어본다. 팀장도 바쁘므로 답장이 늦고, 그때마다 다시 쓰고 또 물어본다. 승인받기까지 수주일이 소요되고, 본인이 서툰 것 같은 죄책감이 생긴다.

**R2 평가자** — 팀장은 직원의 KR을 검토할 때 9개 체크 항목(측정 가능성, 통제 가능성, 도전성, 현실성, 명확성 등)을 일일이 수동으로 검토해야 한다. 하지만 이게 주 업무가 아니라 자동화되거나 체계화되지 않은 일이라, 규칙 위반 KR에 대해 "왜 안 되나"라는 일관된 근거를 제시하기 어렵다. 결국 코칭도 형식적이 되고, 직원도 "왜 안 되는지" 이해 못 한 채 받아들인다.

**R3 인사담당자** — 인사팀 과장은 혼자서 전사 300명의 KR을 형식적으로만 검토한다. 통계를 내거나 위험한 KR을 사전에 식별할 역량이 없다. 평가위원회에서 "이 KR이 이상하지 않나"라는 지적이 나올 때쯤이면, 이미 늦었다. 분쟁이 발생하고, 평가의 공정성이 의심된다.

### 차별적 관점: "규칙 > 자동화, 협업 > 중앙화"

이 상황을 푸는 여러 방법이 있다. ML로 자동 위험 식별을 한다? 중앙 DB에 모든 조직이 데이터를 올린다? 하지만 OKR Lens는 다른 선택을 한다.

1. **규칙 기반 검증** — ML 예측이 아니라 투명한 규칙으로 검증한다. 왜인가? 조직이 OKR을 처음 운영하기 때문이다. 직원과 팀장이 "왜" 안 되는지 명확히 이해할 필요가 있다. 불가해한 AI 점수보다 "너의 KR은 ①② 항목에서 위반"이라는 명확한 피드백이 신뢰를 만든다.

2. **필드 소유권 기반 협업** — 중앙 DB가 아니라 한 곳의 JSON 데이터에서 3역할이 각자의 필드를 소유한다. R1은 KR 초안을, R2는 승인 여부와 코칭을, R3는 위험도 판정을 담당한다. Git의 필드 소유권 개념으로 충돌을 최소화하면서도, 모두가 같은 데이터를 본다. 투명성 + 협업의 균형이다.

3. **AI는 초안만** — AI는 강력하지만 비싸다. Claude API를 R1의 초안 생성에만 써서 비용을 절감하고, 검증은 규칙 엔진이 한다. 이렇게 하면 스케일할 때도 비용이 부담스럽지 않다.

### 성공의 모습

**R1의 변화**: 처음엔 30분을 들여 KR을 손으로 쓰고 팀장에게 물어본다. 이제는? 5분 안에 5개 요소(업무, 제약, 협업 관계)를 입력하고 AI가 4가지 초안을 낸다. "더 도전적으로"라는 한두 줄 피드백으로 수정안을 받고, 마음에 들면 "확정"을 누른다. 1시간 내에 완료. 팀장의 답장을 기다릴 필요가 없다. 그리고 승인률이 높다. 자신감이 생긴다.

**R2의 변화**: "이 KR 뭔가 이상한데"라는 느낌만 있던 팀장이, 이제는 버튼 하나를 누르면 "① 측정 불가능 ② 외부 의존성 있음"처럼 명확한 지적을 할 수 있다. 직원에게 "다시 쓰세요"가 아니라 "측정 방법을 구체화하세요"라고 코칭할 수 있다. 시간은 5분. 근거는 명확하다.

**R3의 변화**: 평가위원회 1주일 전, 자동으로 "위험도 높음" KR 15개가 태깅된다. 각각의 원인이 명시되어 있다("측정 불가능", "신기술 과의존" 등). 과장은 이 15개에 대해 팀장들에게 사전 코칭을 한다. "이 KR을 다시 쓸까요, 아니면 이렇게 진행할까요?" 평가위원회에서 분쟁이 발생하지 않는다. 평가의 공정성에 의문이 제기되지 않는다.

## 🧠 Core Assumptions & Strategic Bets

### 이 제품이 거는 가설

이 제품이 성공하려면 다음 세 가지를 믿어야 한다:

**1. "불완전한 KR이 분쟁의 근본 원인이다"**
- OKR 분쟁의 대부분은 "왜 이 KR로 평가받아야 하나?"가 아니라 "이 KR이 맞는 건가?"에서 비롯된다.
- 검증되지 않은 채 진행된 KR이 평가 시즌에 불거진다.
- 따라서 사전에 명확한 규칙으로 검증하는 것이 분쟁 예방의 핵심이다.

**2. "투명한 규칙 기반 검증이 신뢰를 만든다"**
- ML이나 AI 점수보다 "너는 ① 측정 불가능 ② 외부 의존성"이라는 명확한 이유가 조직 신뢰를 만든다.
- 특히 OKR을 처음 운영하는 조직에서는 더욱 그렇다. 이해할 수 있어야 받아들인다.
- 따라서 자동화가 아니라 일관된 규칙으로 가는 것이 옳다.

**3. "3역할이 같은 데이터를 보면 협업이 증폭된다"**
- R1이 쓴 초안 → R2가 검증 → R3가 위험도 판정하는 과정에서, 각 역할이 필드 소유권을 명확히 하면 충돌이 줄고 신뢰가 만들어진다.
- 중앙 DB 없이 JSON + git으로도 충분하고, 오히려 투명성이 높아진다.
- 따라서 DB 의존성 없이 빠르게 시작하고, 협업 신뢰를 먼저 구축할 수 있다.

## 🔍 Problem Statement

### R1 피평가자
- **문제**: OKR 경험 있지만 KR 작성 자신감 부족
  - 측정 가능성 판단 어려움
  - 도전성/현실성 균형 파악 어려움
  - 다듬는 방법 막막함
  - 매번 평가자에게 물어봐야 함 (승인 지연)

### R2 평가자
- **문제**: 복잡한 OKR 기준 준수 검증 어려움
  - 9개 체크 항목 수동 검토 필수 (시간 소모)
  - 규칙 위반 KR에 대한 반려 근거 부족
  - 코칭 기록 방법 불명확 → 형식적 코칭만 가능

### R3 인사담당자
- **문제**: 전사 KR 일관성 검토 불가능
  - 인력 1명으로 형식적 검토만 가능
  - 이의제기 위험 KR 사전 식별 도구 없음
  - 평가 분쟁 예방 불가

## 💡 Solution Overview

### 핵심 아이디어
**데이터 한 곳, 3역할 협업** — JSON 기반 중앙 저장소 + 규칙 기반 진단 엔진 + Claude AI

```
┌─────────────────────────────────────────────┐
│       src/data/krs.json (공유 저장소)        │
│  ┌──────────┬──────────┬──────────────────┐ │
│  │ R1 작성  │ R2 검증  │ R3 자동 진단     │ │
│  │ (필드)   │ (필드)   │ (필드)           │ │
│  └──────────┴──────────┴──────────────────┘ │
└─────────────────────────────────────────────┘
         ↓
    진단 엔진 (shared/diagnosticEngine.js)
    - 9개 체크 자동 평가
    - 위험도 + 위험 유형 계산
    - 사전 코칭 제안
```

### 기술 철학
- **DB 불필요**: JSON + git (협업 최적화)
- **필드 소유권 명확**: git 충돌 최소화
- **API 선택적 사용**: R1만 Claude API (비용 효율)
- **규칙 기반 엔진**: 투명성 + 재현성 높음

## ✨ Key Features by Role

### **R1 피평가자 — AI KR 작성·정제 Copilot**

#### Feature 1: KR 초안 생성 (자동)
- **입력**: 업무 설명, 제약, 협업 관계 (자유형 텍스트)
- **처리**: Claude API 호출
- **출력**: KR 초안 4종 (각기 다른 관점)
- **UX**: "생성 중..." → 4가지 선택지 표시 → 클릭 선택

#### Feature 2: KR 대화형 정제
- **피드백**: "더 도전적으로", "측정 방법 구체화" 등 자유형 입력
- **처리**: Claude API 재호출 (컨텍스트 유지)
- **출력**: 수정안 + 개선 포인트 명시
- **반복**: 확정될 때까지 반복 가능

#### Feature 3: 확정 & 제출
- **액션**: "최종 확정" 버튼 클릭
- **저장**: src/data/krs.json에 저장 (승인상태=제출)
- **확인**: "제출되었습니다" 메시지 + 평가자 조회 가능 상태

---

### **R2 평가자 — OKR 검토·승인**

#### Feature 1: 메인 대시보드
- **표시**: 팀원 KR 승인률 2~3개 카드
- **액션**: 우선 검토 필요한 KR 강조

#### Feature 2: OKR 검토·승인
- **뷰**: 팀원 OKR 아코디언 (Objective → KR 전개)
- **AI Validation 버튼**: 9개 체크 자동 평가
  - 위반 항목 명확히 표시
  - 위험도 판정 (🟢 하 / 🟡 중 / 🔴 상)
- **승인/반려**: 상태 저장 + 이유 입력 (필수)
- **코칭 메모**: 자유형 입력 (선택)

#### Feature 3: 코칭 기록 (선택)
- **메모**: 자유형 코칭 내용 저장
- **추적**: KR별 코칭 이력 조회 가능

---

### **R3 인사담당자 — 위험 KR 식별 & 사전 대응**

#### Feature 1: 위험 자동 태깅 + 사전 코칭 리스트 (필수)
- **로드**: src/data/krs.json에서 KR 자동 읽기
- **진단**: 진단 엔진 호출 → 위험도·유형 자동 계산
- **필터**: 위험도(🔴 상/🟡 중/🟢 하) 체크박스
- **상세 패널**: 
  - 원문 + 9개 체크 결과 표시
  - 위험 유형 + 권고사항
  - R2의 평가자 피드백 링크

#### Feature 2: 업무군 도출·확정 (필수)
- **1차 분류**: 엔진이 9가지 시드 업무군으로 자동 분류
- **조정**: 인사담당자가 추가/병합/재지정
- **확정**: "확정" 버튼으로 고정 (향후 데이터 분석용)

#### Feature 3: 난이도·품질 분포 (선택 - 시간 있으면)
- **차트**: 상/중/하 분포 바 차트
- **2축 시각화**: x=업무군, y=난이도별 KR 수
- **목표**: 조직 KR 포트폴리오 리스크 시각화

## 🔧 Technical Architecture

### 프로젝트 구조

```
okr-lens/
├── .env.local                     # 환경 변수 (API 키)
├── package.json                   # 공유 의존성
├── next.config.js                 # Next.js 설정
│
├── app/
│   ├── r1-employee/               # R1 페이지
│   │   └── page.jsx               # KR 작성·정제 인터페이스
│   ├── r2-evaluator/              # R2 페이지
│   │   └── page.jsx               # KR 검토·승인 인터페이스
│   └── r3-hr/                     # R3 페이지
│       └── page.jsx               # 위험 진단·코칭 인터페이스
│
├── shared/                        # 공유 모듈 (3역할)
│   ├── types.js                   # KR 스키마 정의
│   ├── diagnosticEngine.js        # 위험 태깅 엔진 (9개 체크)
│   ├── dataAccess.js              # JSON 읽기/쓰기 추상화
│   └── mockDataset.json           # 초기 mock KR 10개
│
└── src/data/
    └── krs.json                   # ⭐ 공유 KR 데이터 저장소
```

### 기술 스택

| 계층 | 기술 | 이유 |
|------|------|------|
| **프레임워크** | Next.js 14 (App Router) | 빠른 개발 + 배포 용이 |
| **스타일** | Tailwind CSS | 유틸리티 우선 → 빠른 UI 구현 |
| **차트** | Recharts | 경량 + 반응형 (R3 분포도) |
| **상태관리** | React Context + useState | 시스템 복잡도 낮음 (DB 불필요) |
| **AI** | Claude API (claude-sonnet-4-6) | 자연어 처리 + 비용 효율 |
| **데이터** | JSON + git | 협업 최적 + 버전 관리 |
| **배포** | Vercel 또는 로컬 | 빠른 배포 + 모니터링 |


### 진단 엔진 (Diagnostic Engine)

**9개 체크 항목**:

```
① 수치로 측정 가능한가? 
② 외부 의존성이 없는가? (통제 가능한가?)
③ 도전적인가? (유지형이 아닌가?)
④ 시간 내 현실적으로 달성 가능한가?
⑤ 명확한 언어인가? (팀원이 이해 가능한가?)
⑥ 다른 팀 KR과 겹치지 않는가?
⑦ 신기술/새 도구에 과도히 의존하지 않는가?
⑧ 단순 건수형이 아닌가? (질이 있는가?)
⑨ 평가자가 확인할 수 있는가? (증거가 있는가?)
```

**위험도 판정**:
- 체크 위반 0~1개 → 🟢 **하** (코칭 불필요)
- 체크 위반 2~3개 → 🟡 **중** (주의, 보완 권고)
- 체크 위반 4개 이상 → 🔴 **상** (높은 위험, 사전 코칭)

## 📊 Success Criteria — User-Centric View

### R1 피평가자 (목표: "30분 → 10분, 자신감 ↑")
**사용자가 체감해야 할 변화**:
- [ ] **속도**: 요구사항 입력 5분 → AI 초안 4종 생성 → 피드백 정제 2회 → 확정 버튼 → 총 10분 이내 (기존 30분)
- [ ] **자신감**: "혼자서 완성했다" 느낌 (기존: "팀장 승인 기다림")
- [ ] **승인률**: 첫 제출에서 반려율 <20% (기존: 반려→재작성→재제출 반복)

**기술 성공 기준** (위의 사용자 체감을 가능하게 하는):
- [ ] 업무·제약·협업 입력 → KR 초안 4종 생성 (**Claude API 호출**)
- [ ] 자연어 피드백("더 도전적으로") → 수정안 생성 (**재호출**)
- [ ] 확정 → src/data/krs.json에 저장 (승인상태=**제출**)

---

### R2 평가자 (목표: "왜인지 알고 코칭하기")
**사용자가 체감해야 할 변화**:
- [ ] **속도**: KR 검토 5분 (기존: 수동 9개 체크 15분)
- [ ] **근거**: "다시 쓰세요" 대신 "①번 항목(측정 불가능)을 보완하세요" 코칭 (일관성 ↑)
- [ ] **코칭 품질**: R1이 "아, 이래서 안 되는구나" 이해 → 재작성 만족도 ↑

**기술 성공 기준**:
- [ ] src/data/krs.json에서 KR 읽기 (R1 제출한 것)
- [ ] "AI Validation" 클릭 → **9개 체크 자동 계산** → 위험도 출력
- [ ] "승인" 클릭 → 파일 업데이트 (승인상태=**승인**)

---

### R3 인사담당자 (목표: "사전 예방, 분쟁 예방")
**사용자가 체감해야 할 변화**:
- [ ] **예측성**: 평가위원회 1주일 전 "위험도 높음" KR 자동 태깅 → 사전 코칭 리스트 준비
- [ ] **분쟁 예방**: 위험 KR을 팀장과 함께 사전 조율 → 평가위원회에서 분쟁 0건
- [ ] **공정성**: 평가 결과에 이의제기율 <5% (기존 수치 미확인, 목표 설정)

**기술 성공 기준**:
- [ ] src/data/krs.json 로드 → **위험도·유형 자동 태깅**
- [ ] 위험도 필터(🔴 상/🟡 중/🟢 하) **동작**
- [ ] 상세 패널: 원문 + 9개 체크 + 위험 유형 + 권고 **표시**

---

### 통합 성공 (End-to-End Handoff)
- [ ] **R1 제출 → R2 승인 → R3 위험 리스트 반영** (연쇄 동작 ✓)
- [ ] **데이터 정합성**: R2에서 본 값 = R3에서 표시하는 값 (동일 ✓)
- [ ] **신뢰 형성**: 3역할 모두 "이 시스템이 우리가 필요했던 거다"라는 피드백
