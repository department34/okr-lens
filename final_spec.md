# OKR Lens — 최종 MVP Spec (v3.0)

> **최종본 v3.0** | 2026-06-24
> 3개 Spec 파일(R1, R2, R3) 통합 분석 + 최적화
>
> **핵심**: R3의 현실적 10일 일정 + R2의 명확한 아키텍처 + R1의 구체적 산출물 + R3의 협업 최적 데이터 관리

---

## 0. 개요 · 통합 · 시연 연계 (3역할 공통)

**제품 한 줄**: 2026 OKR 기반 개인평가제도를 처음 운영하는 IT 조직을 돕는 3역할 도구 — KR을 잘 쓰게(피평가자) → 잘 검토·승인하게(평가자) → 위험을 사전 식별하게(인사담당자).

**3역할 / 동시 개발**: R1 피평가자 · R2 평가자 · R3 인사담당자를 **담당자 1인씩, 3명이 동시 개발**.

**개발 기간**: **10일 (실제 일정)** — 3명이 퇴근 후 1~2시간씩 협력 시 약 3주 소요

**범위 우선순위**: **R1 완벽 > R2 반 정도 > R3 간단하게** (시간 부족 시 우선순위 따라 단계적 축소)

**시연 스토리 (핸드오프)**:

1. **R1 피평가자** — 업무·제약·협업 입력 → AI가 KR 초안 4종 생성 → 대화형 정제 → 확정·제출
2. **R2 평가자** — 제출된 KR 조회 → AI Validation(규칙 검증) → 승인/반려
3. **R3 인사담당자** — 승인된 KR 로드 → 위험 자동 태깅 → 사전 코칭 리스트 제시

---

## 1. 문제 — 누구의 어떤 불편을 푸는가

**R1 피평가자**: OKR 경험은 있으나 KR 작성 자신감 부족. 측정 가능성·도전성 판단 어렵고, 다듬는 방법 막막, 매번 평가자에게 물어보기 부담.

**R2 평가자(팀장)**: 복잡한 OKR 작성 기준(난이도 9항목 체크, KR 4~6개 등)으로 승인 시 규칙 위반 즉시 판단 어려움. 코칭 기록 작성 방법 불명확.

**R3 인사담당자**: 전사 KR 일관 검토 필요하나 인력 1명이라 형식적 검토만 가능. 이의제기 위험 KR 사전 식별 도구 부재.

## 3. 핵심 기능 (MVP)

### **R1 피평가자 — AI KR 작성·정제 Copilot**

**Feature 1**: KR 초안 입력 → AI 4종 초안 생성

- 업무·제약·협업 입력 → Claude API 호출 → 4가지 KR 후보 생성

**Feature 2**: KR 대화형 정제

- 자연어 피드백("더 도전적으로") → 수정안 + 개선 포인트 제시

**Feature 3**: 확정 → 평가자 제출

- 최종 KR 저장 (공유 데이터, 승인상태=제출)

> **현실성**: Feature 1·2·3 모두 10일 안에 완성 가능 ✅

---

### **R2 평가자 — OKR 검토·승인**

**Feature 1**: 메인 대시보드 (간단하게)

- OKR 승인률 카드 1~2개 (캘린더 제외)

**Feature 2**: OKR 검토·승인

- 팀원 OKR 아코디언 뷰
- **AI Validation 버튼**: 9개 체크 항목 자동 평가 → 위험도 판정
- 승인/반려 상태 저장

**Feature 3**: 코칭 기록 (선택)

- 코칭 메모 수기 입력 (AI 자동 제안 제외)

> **현실성**: Feature 1·2 필수, Feature 3 여유 있으면 추가 ✅

---

### **R3 인사담당자 — 위험 KR 식별**

**Feature 1 (우선순위 1)**: 위험 자동 태깅 + 사전 코칭 리스트

- KR 로드 → 진단 엔진 호출 → 위험도·유형 자동 태깅
- 위험도(상/중/하)·유형·부서 필터링
- 상세 패널: 원문 + 9개 체크 + 위험 유형 + 권고

**Feature 2 (우선순위 2)**: 업무군 도출·확정

- 엔진이 9가지 시드 업무군으로 1차 분류
- 인사담당자가 추가/병합 후 "확정"
- 부서별 구성 시각화

**Feature 3 (우선순위 낮음)**: 난이도·품질 분포

- 상/중/하 분포 + 2축 차트
- ⚠️ 시간 부족 시 생략 가능

> **현실성**: Feature 1·2 필수, Feature 3 선택 ✅

## 5. 위험 태깅 엔진 (R2·R3 공유)

### **9개 체크 항목**

```
① 수치로 측정 가능한가?
② 외부 의존성이 없는가? (통제 가능한가?)
③ 도전적인가? (유지형이 아닌가?)
④ 시간 내 현실적으로 달성 가능한가?
⑤ 명확한 언어인가? (팀원이 이해 가능한가?)
⑥ 다른 팀 KR과 겹치지 않는가?
⑦ 신기술/새 도구에 과도히 의존하지 않는가?
⑧ 단순 건수형이 아닌가? (질이 있는가?)
⑨ 평가자가 확인할 수 있는가? (증거가 있는가?)
```

### **위험도 판정**

```
위반 항목 수 (체크=0)    →  위험도
0~1개                  →  🟢 하 (코칭 불필요)
2~3개                  →  🟡 중 (주의, 보완 권고)
4개 이상               →  🔴 상 (높은 위험, 사전 코칭)
```

### **위험 유형 매핑**

```
① → 측정모호 (C05)
② → 통제불가 (C11)
③ → 도전성저하 (현상유지)
④ → 현실성낮음
⑤ → 표현모호
⑥ → 공모형
⑦ → 신기술의존 (C12)
⑧ → 건수형지표 (C06)
⑨ → 확인불가 (C01)
```

## 7. Git 워크플로우 & 협력 프로세스

### **브랜칭 전략**

```
main (배포·시연 기준)
  ├─ develop (통합 브랜치)
  │  ├─ feature/r1-... (R1 작업)
  │  ├─ feature/r2-... (R2 작업)
  │  └─ feature/r3-... (R3 작업)
```

### **PR & 리뷰**

```
매일 저녁:
  각자 진도를 PR로 올림 (feature 브랜치 → develop)
  커밋 메시지: "feat(r1): KR 초안 생성 로직 추가"

매일 아침:
  3명 중 1명이 PR 리뷰·승인 (코드 리뷰 포인트 참고)
  → 병합

공유 파일 변경 (shared/types.js, shared/diagnosticEngine.js):
  → 3명 모두 Approve 필수 (Day 3~4 스키마 검증 이후)
  → 진도 회의에서 함께 검증
```

### **코드 리뷰 포인트**

- 공유 스키마(KRSchema) 준수 여부
- 진단 엔진 로직 일관성 (R2·R3)
- src/data/krs.json 데이터 흐름 정합성

### **진도 동기화 (비동기 팀)**

**주 2회, 30분 미팅**:

```
수요일 (진도 + 블로커):
  - 각자 어디까지 했는가 (Day 기준)
  - 막힌 부분 있는가? (기술·스키마)
  - 내일 우선순위

금요일 (통합 검증 + 다음 주):
  - 이번 주 구현 기능 테스트 (최소 경로)
  - R1→R2→R3 데이터 흐름 검증
  - 다음 주 Day 목표
```

### **긴급 소통**

```
Slack #dev-alert: 기술 블로커 (목표: 1시간 내 해결)
Slack #design: 데이터 계약 변경 (3명 동의 후 반영)
```

## 8. 성공 기준 — 시연에서 무엇이 되면 성공인가

**R1 피평가자**

- [ ] 업무·제약·협업 입력 → KR 초안 4종 생성 (Claude API 호출)
- [ ] 자연어 피드백("더 도전적으로") → 수정안 생성 (재호출)
- [ ] 확정 → src/data/krs.json에 저장 (승인상태=제출)

**R2 평가자**

- [ ] src/data/krs.json에서 KR 읽기 (R1 제출한 것)
- [ ] "AI Validation" 클릭 → 9개 체크 자동 계산 → 위험도 출력
- [ ] "승인" 클릭 → 파일 업데이트 (승인상태=승인)

**R3 인사담당자**

- [ ] src/data/krs.json 로드 → 위험도·유형 자동 태깅
- [ ] 위험도 필터(상/중/하) 동작
- [ ] 상세 패널: 원문 + 9개 체크 + 위험 유형 + 권고 표시

**통합 (핸드오프)**

- [ ] R1 제출 → R2 승인 → R3 위험 리스트 반영 (연쇄 동작)
