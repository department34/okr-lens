## 🎯 의사결정 포인트 (선택 기준)

### **1. 일정 현실성 — 가장 중요**

#### **new_spec_all.md (3주)**
- ✅ **장점**:
  - 여유 있는 일정으로 완성도 높은 시연 가능
  - 팀 협력 + 정기점검 + 통합검증에 충분한 시간
  - 예기치 않은 이슈 대응 가능
  - DB 마이그레이션 로드맵까지 포함

- ❌ **단점**:
  - **3주는 누적 51시간 (3명이 평균 17시간씩 투입)**
  - 퇴근 후라면: 주당 5시간 × 3주 = 15시간 (부족)
  - **풀타임이라도 "3주 동안 다른 업무 없다"는 가정 필요**
  - 현실: "2주 안에 끝내고 싶으면?" → 압박

#### **final_spec.md (10일)**
- ✅ **장점**:
  - **현실적**: "퇴근 후 1-2시간 × 10일" 명시
  - 빠른 피드백 루프 가능 (주말 테스트)
  - 긴급 상황에 빠르게 대응
  - 모멘텀 유지 (3주는 피로 누적)

- ❌ **단점**:
  - Day별 할당량 촘촘함 (Day 3-4에 최소 경로 완성)
  - 여유 시간 거의 없음
  - 버그 발견 시 빠른 대응 필수
  - 야근 가능성 높음

**→ 평가**: **현실적으로는 final_spec.md의 10일이 더 정확함**. 실제 팀이 "3주 풀타임"인지 "퇴근 후"인지에 따라 결정.

### **2. 데이터 관리 방식**

#### **new_spec_all.md**
```
shared/mockDataset.json (공유, 모두 읽음)
├─ R1: 저장
├─ R2: 업데이트 (승인상태, 위험도)
└─ R3: 읽기
```
- **장점**: 중앙 집중식, 데이터 일관성 보장
- **단점**: 여러 명이 동시에 수정할 때 git 충돌 가능 (필드 소유권으로 완화)

#### **final_spec.md**
```
src/data/krs.json (중앙 집중, git 필드 소유권)
├─ R1 소유: objective, kr, 측정방법, 체크, 난이도
├─ R2 소유: 평가자, 승인상태, 반려사유
├─ R3 소유: 업무군_확정, 코칭제안
└─ 엔진: 위험분석 (자동)
```
- **장점**: 필드 소유권으로 충돌 최소화, **더 현실적**
- **단점**: 없음 (이미 검증된 방식)

**→ 평가**: **final_spec.md의 필드 소유권 방식이 더 안전함**.

## 🎯 **최종 권장사항**

### **→ final_spec.md를 선택하세요**

**이유:**
1. **현실성** (가장 중요): 3주는 실무에서 거의 불가능. "퇴근 후 1-2시간 × 10일"은 달성 가능한 목표.
2. **필드 소유권**: src/data/krs.json의 필드별 소유권 방식이 git 충돌을 최소화 (새로운 팀에는 필수).
3. **빠른 검증**: Day 3-4 최소 경로로 조기에 스키마 오류 발견 가능.
4. **유연성**: 10일 안에 끝나면 좋고, 연장되어도 "1주 더" 정도로 조정 가능.

### **그러나 보충이 필요하면:**
- final_spec.md를 **기본** 문서로 사용
- **필요한 부분만** new_spec_all.md에서 참고:
  - Section 12: 문제해결 가이드
  - Section 7: 위험 태깅 엔진 상세 (9개 체크 + 위험 유형)
  - Section 8.2: Day별 구체적 시간 배분
