# OKR Lens — 통합 MVP Spec (피평가자 · 평가자 · 인사담당자)

> 통합본 v1.4 | 2026-06-22 | 작성: 신승희 / 인사노무팀
> 원본: 피평가자용 spec(R1) · 평가자용 spec v1.0(R2) · 인사담당자용 spec v1.6(R3) 통합
> 시연 스토리: **피평가자 → 평가자 → 인사담당자** (인사담당자 클로징)

---

## 0. 개요 · 통합 · 시연 연계 (3역할 공통)

**제품 한 줄**: 2026 OKR 기반 개인평가제도를 처음 운영하는 IT 조직을 돕는 3역할 도구 — KR을 잘 쓰게(피평가자) → 잘 검토·코칭·승인하게(평가자) → 전사 품질·위험을 보게(인사담당자).

**3역할 / 동시 개발**: R1 피평가자 · R2 평가자 · R3 인사담당자를 **담당자 1인씩, 3명이 동시 개발**.

**시연 스토리 (핸드오프)**:

1. **R1 피평가자** — 업무·제약·협업(상위 Objective는 선택) 입력 → AI가 KR 초안 4종 생성 → 대화형으로 정제 → (멀티)AI 교차 검증 → 확정·제출.
2. **R2 평가자** — 제출된 KR을 받아 룰 위반 자동 점검(AI Validation)·코칭 후 승인.
3. **R3 인사담당자 (클로징)** — 승인된 KR 포함 전사 KR을 집계 → 이의제기 위험 KR 사전 식별 → 업무군 도출·확정 → 구성·난이도/품질 분포 제시. "한 사람의 KR이 전사 그림의 일부가 되는" 마무리.
   - 1·2단계에서 등장한 KR을 **공유 mock 전사 데이터셋에 포함**시켜 R3 화면에서 다시 보이게 한다 → 스토리 연결.

**공유 KR 데이터 계약** (3역할 공통 스키마, 캘리브레이션 데이터 컬럼과 정합):

```
{ id, 부서, 담당자, 직급,
  objective, kr, 측정방법, 달성도: {S,A,B,C,D}, 난이도,
  체크: [①..⑨],                         // 난이도 체크리스트
  평가자, 승인상태,                        // R2가 갱신 (제출/승인/반려)
  업무군, 경계, 위험태그:[], 위험도 }       // 진단 엔진 1차 부여 → R3가 업무군 확정
```

R1 생성·제출 → R2 검토·승인(승인상태 갱신) → R3 읽기·업무군 확정. **각 역할은 자기 화면만 구현**하고, 다른 역할과는 이 데이터로만 연계.

**공유 진단 엔진**: 위험 태깅 + 업무군 1차 분류 룰 모듈은 **R2(승인 시 위반 점검)와 R3(전사 위험 식별)가 공유** — 판정 일관·중복 구현 방지. (캘리브레이션 analyze.py 규칙 이식)

**구동 환경**: **로컬 서버(localhost) 구동**을 기본(AXHub 사용 불가 가능성 대비). 스택을 **Next.js로 통일**했으므로 통합 방식은 **단일 Next.js 앱에 역할별 라우팅**(`/employee`·`/evaluator`·`/hr`)으로 한다 — 한 번의 `npm run dev`로 3역할을 모두 띄워 시연 핸드오프가 자연스럽다. 모든 데이터는 로컬 보관·외부 전송 없음. AXHub 가용 시 코드 변경 최소로 사내 빌드 이관.

## 1. 문제 — 누구의 어떤 불편을 푸는가

**R1 피평가자**: OKR 운영 경험은 있으나 KR 문장을 직접 쓰는 자신감이 부족하다. 무엇을 입력해 시작할지, 자기 KR이 측정 가능한지·도전적인지 스스로 판단하기 어렵고, 고치려 해도 어떻게 다듬을지 막막하며, 매번 동료·평가자에게 묻기는 부담스럽다.

**R2 평가자(팀장)**: OKR 작성 기준(운영/전략혁신, 난이도 체크 9항목, KR 4~6개 등)이 복잡해 승인 시 룰 위반을 즉각 판단하기 어렵다. 월 1회 1on1·분기 Align 기록을 의무 등록해야 하나 무엇을 남길지 막막하다. 연말에 팀원별 KR 달성도·코칭 이력을 종합 판단하기 어렵다.

**R3 인사담당자**: 전 직원 KR을 일관 기준으로 검토·코칭해야 하나 인력은 1명이라 형식적 검토에 그친다. 평가위원회 전 이의제기 위험 KR을 미리 거를 도구가 없다. 전사·부서 지표가 어떤 업무 성격(매뉴얼·자동화·개발 등)에 몰렸는지, 측정가능성·도전성·난이도가 어떻게 분포하는지 한 화면에서 못 본다.

## 3. 핵심 기능 (MVP) — 역할별 2~3일 범위

### R1 피평가자 — AI KR 작성·정제 Copilot (단계형 위저드)

작성자가 KR을 처음부터 끝까지 만들고 다듬어 평가자에게 제출하는 흐름. 화면 단계: **① 목표 선택 → ② KR 초안 입력 → ③ 초안 확인 → ④ 대화형 정제 → ⑤ 최종 검토 → ⑥ 완료(제출)**.

**Feature 1. KR 초안 입력 → AI 4종 초안 생성**

- 업무 설명 · 제약 조건 · 협업 대상을 입력하면(상위 Objective는 선택 입력) AI가 현실적인 KR 초안 **4가지**를 생성한다.

**Feature 2. KR 대화형 정제**

- "데이터 품질 기준을 더 구체적으로", "목표를 90점으로 올려 더 도전적으로" 같은 **자연어 피드백** → AI가 수정안 + 개선 포인트 제시, "다른 제안 보기 / 이 내용으로 적용".

**Feature 3. 최종 검토 — (멀티) AI 교차 검증**

- 정제된 KR을 강점·보완·평가위원회 리스크(측정/과도한 목표/외부 의존성)·추천 수정안으로 교차 검토하고, **AI 종합 추천안 + 버전 히스토리**를 제시. 항목별 "이 수정안 반영 / 일부 인용 / 무시".
- ※ **멀티 모델(Claude·Gemini·OpenAI) 동시 교차 검증은 외부 다중 API가 필요해 폐쇄망·2~3일 MVP 범위 밖.** MVP는 **단일 모델(Claude) 또는 룰 기반 단일 검토**로 구현하고, 멀티 AI는 인터넷 가용 시연/Phase 2 확장으로 둔다.

**Feature 4. 확정 → 평가자 제출**

- 최종 KR을 편집·확정하면 공유 KR 데이터(승인상태=제출)로 **평가자(R2)에게 넘어간다** → 시연 핸드오프의 시작점.

> MVP 핵심(2~3일): Feature 1·2 + 4. Feature 3은 단일 모델/룰 기반으로 축소 구현.

### R2 평가자 — OKR Lens

**Feature 1. 메인 대시보드**

- 이번 달 코칭 일정 캘린더 (예정 미팅 D-day 표시).
- 이번 달 현황 요약 카드 3개 (OKR 승인률 / 코칭 등록률 / 평가 완료율).
- *(랭킹 테이블·이모지 등급·최하 팀원 알림 배지 → 이번 시연 제외, 4절 참조)*

**Feature 2. AI 코칭 지원**

- 팀원별 KR 달성 현황 카드 조회.
- 코칭 기록 수기 입력 + **AI 제안 버튼**(Claude API) — 클릭 시 기존 내용 덮어쓰기 Confirm 팝업 → 확인 시 대체.
- 1on1/Align 미팅 일정 등록 및 이력 조회.
- 코칭 메모 AI 배지(🤖 AI 자동입력 / ✏️ 수정됨).

**Feature 3. OKR 검토·승인**

- 팀원 OKR 아코디언 뷰(Objective → KR → Initiative).
- **AI Validation 버튼**: 2026 제도 기준 위반 자동 감지 (KR 개수 4~6개, 난이도 상 비율 50% 이내, 수치 측정 가능 여부 등).
- 승인 / 반려(사유 입력), OKR 수정 요청 승인 루프(수정 이력 보관).

### R3 인사담당자 — 전사 OKR 품질/구성 대시보드

**Feature 1. 이의제기 위험 KR 자동 식별 → 사전 코칭 리스트 (최우선)**

- 전 직원 KR 로드 시 룰 엔진이 KR별 **업무군**과 **이의제기 위험 유형·위험도**를 자동 태깅.
- 위험 유형(캘리브레이션 진단 체계와 동일): 공모형 *(선제안 등 전사 공통지표는 제외)* · 통제 불가 외생변수(SLA·만족도·매출·납기) · 지표 사용불가(자격·교육 둔갑) · 측정 모호 · 도전성 저하(유지형) · 단순 건수형 · 난이도 ④⑥의존(⑦⑧ 없이 신기술·전사만으로 "상").
- 위험도(상/중/하)·유형·부서 필터·정렬한 "사전 코칭 대상" 리스트 + 권고 1줄. KR 클릭 시 상세 패널(원문 + 위험 유형 + 체크①~⑨ + 왜 위험·권고).
- **위험 태그 = "코칭 후보(이상 징후 트리거)", 위반 아님** — 확정 아님, 사전 점검·보완용.

**Feature 2. 업무군 도출·확정 → 지표 구성 분석 (필수, 평가자 참고자료)**

- 업무군은 고정 목록이 아니라 **모인 지표를 분석해 도출·확정**한다.
  1. 1차 도출 — 엔진이 KR·측정·A등급 산출물을 분석해 분류. **시드 9종**(개발/요건·데이터/정합성·자동화·성능/튜닝·장애/운영안정·정시/납기/처리율·매뉴얼/표준/문서·보안/권한/개인정보·고객/사업기여/제안, *향후 확장 가능*)에서 출발, 안 맞는 지표는 '경계·미분류'로 모음.
  2. 검토·확정 — 인사담당자가 업무군 추가·이름변경·병합 + 경계/미분류 재지정 후 **"확정"**. 확정 분류가 분석 기준.
- 확정 분류로 전사·부서·팀별 **업무군별 지표 건수·비율** 시각화 → "매뉴얼이 많은지, 자동화가 많은지, 개발 중심인지" 한눈에. 부서 간 구성 비교.
- **목적**: 평가자 판단용 참고자료(구성 정보). 난이도 일관성 검증 아님(같은 업무군이라도 시스템·환경에 따라 난이도는 정당하게 달라짐).

**Feature 3. 난이도·품질 분포 대시보드 (업무군과 별개 차원)**

- 난이도 상/중/하 분포(전사·부서별, 업무군과 교차하지 않음).
- 품질 2축 분포 — **측정가능성·도전성**(양호/주의/위험), 부서 편차. *(정렬성 제외 — 상위 Objective 선택 입력·팀 OKR 미정의, 4절)*
- 요약 카드(전사 KR 수 / 위험 KR 수·비율 / 코칭 대상 수 / 경계·미분류 수), 차트 클릭 시 드릴다운.

## 4. 제외 범위 — 이번엔 안 하는 것

**공통**

- 사용자 인증·권한 관리 (프로토타입 단일 계정 고정)
- 실제 DB·인사시스템/ERP 연동 (JSON/엑셀 mock 대체), 전년 데이터 자동 축적
- 이메일/슬랙 알림 (앱 내 배지·리스트로 대체)
- 다른 역할 화면 자체 구현 (각 담당 몫, 공유 KR 데이터로만 연계)

**R1 피평가자**

- **품질 진단(실시간 3축 점수: 측정가능성·도전성·통제가능성) 화면 — 이번 시연 제외.** KR 초안 생성·대화형 정제·교차 검증에 집중
- **멀티 AI(Claude·Gemini·OpenAI) 동시 교차 검증 — 외부 다중 API 필요, 폐쇄망·MVP 범위 밖.** MVP는 단일 모델/룰 기반 단일 검토, 멀티 AI는 Phase 2(또는 인터넷 가용 시연)
- 작성 이력 영구 저장·버전 관리 (MVP는 세션 내 버전만)

**R2 평가자**

- **⛔ 팀원 실시간 랭킹 테이블·이모지 등급(🏆/🟢/🟡/🟠/🚨) — 이번 시연 제외**
- **⛔ 최하 팀원 코칭 미등록 알림 배지(14일 초과 등) — 랭킹 연동 항목, 이번 시연 제외**
- **⛔ 랭킹 공통 노출(대시보드·코칭 스케줄 팝업·AI 코칭 상단) — 이번 시연 제외**
- 연말 평가 점수 입력 화면, 팀원 평가 이력 조회, 다면/상향평가, 이의제기 처리

**R3 인사담당자**

- **정렬성(상위 목표와의 정렬) 평가 — 스펙 전체 제외(R1 품질 진단·R3 분포 공통). 상위 Objective가 선택 입력이라 정렬 대상이 일정치 않고, 팀(상위) OKR도 미정의. 상위 Objective 상시화/팀 OKR 도입 시 Phase 2**
- 같은 업무군 내 난이도 일관성 자동 검증·난이도 자동 재산정 (시스템·환경 차이로 정당 → 위반 판정 안 함)
- 업무군 완전 비지도 자동 군집(임베딩) — MVP는 시드+경계 노출→확정 방식, Phase 2
- 연말 점수 입력·평가위원회 운영·이의제기 처리 워크플로우

## 5. 성공 기준 — 시연에서 무엇이 되면 성공인가

**R1 피평가자**

- [ ] 업무·제약·협업을 입력하면(상위 Objective는 선택) KR 초안 4종이 생성된다
- [ ] 자연어 피드백("더 도전적으로" 등)으로 KR이 정제된다(수정안 + 개선 포인트 제시)
- [ ] 확정하면 KR이 평가자에게 제출된다(공유 데이터 승인상태=제출)

**R2 평가자**

- [ ] OKR 카드에서 AI Validation 버튼을 누르면 위반 항목이 텍스트로 출력된다
- [ ] 코칭 AI 제안 버튼 → Claude 생성 → Confirm 팝업 후 텍스트 대체된다
- [ ] 승인/반려 동작이 UI에서 작동하고 상태가 변경된다
- [ ] 캘린더에 이번 달 1on1 일정이 표시된다 *(보너스)*

**R3 인사담당자**

- [ ] 전사 KR(공유 mock)을 로드하면 KR별 업무군(1차)·위험 유형이 자동 태깅된다
- [ ] 모인 지표로 업무군이 도출되고, 인사담당자가 추가·재지정 후 "확정"할 수 있다(경계/미분류 포함)
- [ ] 사전 코칭 대상 리스트가 위험도·유형·부서와 함께 출력·필터된다
- [ ] 확정 업무군 기준으로 부서별 구성(건수·비율)이 보인다
- [ ] 난이도 상/중/하 분포와 측정가능성·도전성 분포가 표시된다
- [ ] 공통지표(선제안 등)는 공모형으로 오탐되지 않는다

**통합 (핸드오프)**

- [ ] R1이 작성·제출 → R2가 승인한 KR이 R3 전사 코칭 리스트·분포에 반영되어 나타난다

## 6. 기술 스택

**통합 권장**: 로컬 서버(localhost) 구동, 데이터 로컬 보관. **3역할 모두 Next.js로 통일** — 단일 앱에 역할별 라우팅으로 통합한다. 핵심 진단은 외부 API 없이 동작 → 로컬·폐쇄망 무관.

```
프레임워크:  Next.js 14 (App Router) — 3역할 공통(통일), 단일 앱 + 역할별 라우팅
             (/employee=R1, /evaluator=R2, /hr=R3)
스타일링:    Tailwind CSS
차트:        Recharts (R3 업무군 분포 · 난이도 분포 · 부서 편차)
언어:        JavaScript (JSX) — 프로토타입, TypeScript 미사용
상태관리:    React useState / useContext
데이터:      공유 KR 데이터셋(JSON) + 엑셀 업로드(SheetJS, R3) / R1 가이드·예시는 정적 JS 모듈
진단 엔진:   룰 기반(결정적) JS 모듈 — 위험 태깅 + 업무군 1차 분류. R2·R3 공유
AI:          Anthropic Claude API (claude-sonnet-4-6)
             — R1은 KR 초안 생성·대화형 정제·검토에 **상시 사용**(LLM 의존)
             — R2 코칭 문구 제안, R3 코칭 문구·새 업무군 후보 제안은 선택
             (R2·R3 핵심 진단은 룰 기반이라 API 없이 동작 / R1은 LLM 필요 →
              폐쇄망 이관 시 사내 LLM 대체 필요. 로컬 시연은 인터넷으로 Claude API 사용)
구동·배포:   시연 = 로컬 서버(npm run dev 또는 build 후 정적 서빙). AXHub 가용 시 이관
환경변수:    .env.local (ANTHROPIC_API_KEY, 선택) — .gitignore 필수
```

## 7. 개발 우선순위 (3역할 병렬, 10일 기준)

**⭐ 핵심 변경**: Day 1~10 실제 일정 + Day 3~4 최소 경로 통합(KR 1개 흐름) + R3 우선순위 명확화

```
[Day 1] 오전 (2시간)
  [공통] 공유 데이터 계약·진단 엔진 합의/이식(위험 태깅 7종 + 업무군 9종)
  [공통] 환경 셋업 (Node.js, npm, Claude API 키)

[Day 1~2] 오후 (3시간)
  [공통] 첫 PR 경험 (feature/setup, git 협업 배우기)
  [공통] git 질문 정리 및 완료

[Day 3~4] ⭐ **최소 경로 통합 (가장 중요)**
  [R1] KR 1개만 생성 & 대화형 정제 & 확정·제출 (1일)
  [R2] R1의 KR 1개 검토·승인 (0.5일)
  [R3] 규칙 엔진 구현(기본 버전) + R2의 KR 1개 분류·코칭 (1.5일)
  → [공통] 데이터 정합성 검증 (1시간) = 스키마 오류 즉시 fix

[Day 5~6] 각 역할 고도화 (병렬)
  [R1] 나머지 3종 KR 초안 생성 + 대화형 정제 + 최종 검토(단일 모델/룰 기반) → 완성
  [R2] 메인 대시보드(캘린더 + 현황 카드 3개) + OKR 검토·승인 UI + AI Validation 버튼 → 완성
  [R3] **우선순위 1**: 위험 태깅(Feature 1) 완성
        **우선순위 2**: 업무군 도출·확정 UI(Feature 2) 완성
        사전 코칭 리스트(위험도·유형·부서 필터 + 상세 패널)

[Day 7] 정기점검 (1시간)
  [공통] R2→R3 데이터 연계 검증
  [공통] 업무군 분류 규칙 샘플 테스트 (5~10개 KR)
  [공통] 문구 톤 일관성 점검("위반" ❌ → "코칭 후보" ✓)

[Day 8~9] 추가 기능 + R3 우선순위 관리
  [R1] 작성 이력 영구 저장(localStorage) + 버전 히스토리
  [R2] 팀원별 코칭 이력 조회 또는 간단 랭킹 비교
  [R3] **우선도 낮음**: 난이도·품질 2축 분포 차트(Feature 3) 시작
        ※ 시간 부족하면 Feature 3 제외 가능
        ※ 여유 있으면 공통지표(선제안) 자동 필터링 추가
  [공통] mock 데이터 보강 (15개 → 30개 KR)

[Day 10] 최종 리허설 + 정리 + 시연 준비 (2시간)
  [공통] R1→R2→R3 전체 흐름 최종 테스트
  [공통] 데이터 정합성 재검증
  [공통] UI 오탈자·버그 픽스
  [공통] README 작성(실행 방법, 데이터 포맷)
  [공통] 발표 흐름 리허설(R1→R2→R3)
  [공통] 예상 질문 준비(규칙, 소송 톤, 데이터 확장)
```

**핵심 개선**:
1. **일정 재구성**: Day 1~10으로 명확히 (기존 예시 일정 → 실제 10일)
2. **최소 경로 통합**: Day 3~4에 KR 1개를 R1→R2→R3까지 끝까지 흘려 **스키마 검증 우선**
3. **R3 우선순위 명확화**: Feature 1(위험 태깅) > Feature 2(업무군) >> Feature 3(분포), Day 10에 제외 가능
4. **정기점검 추가**: Day 7에 규칙 검증 시간 명시
